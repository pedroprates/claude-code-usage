CC Usage Tracker · Investigation · 4 July 2026

The number is reliable. The shared file is not.

Claude Code supplies an authoritative account-usage percentage. The tracker’s current collector lets several sessions repeatedly overwrite one global snapshot with differently aged copies of that percentage.

●
Root cause reproducedThe active status-line command uses a last-writer-wins script. Three live Claude sessions changed the menu-bar source from 55% to 75% to 99% within seconds, despite sharing the same reset window.

How the current app works

The Swift app does no token estimation. Its Claude side is a local relay: Claude Code emits rate-limit data, a shell command persists it, and the menu app renders the latest file.

1 · Claude CodeRuns the configured status-line command and passes session JSON on standard input, including rate_limits after an API response.
→
2 · ccstatuslineExtracts 5-hour and 7-day percentages and atomically replaces ~/.claude/cc-usage-tracker/state.json.
→
3 · Menu-bar appUsageStore watches the directory, decodes the file, and rounds the 5-hour value for the macOS menu bar.
Source

Official rate-limit field

rate_limits.five_hour.used_percentage, already calculated by Claude. This is the right raw source.

Storage

One machine-wide file

Every Claude process writes the same path, regardless of session ID or snapshot age.

Presentation

Faithful, not faulty

The Swift UI simply reads and rounds the file. It is accurately displaying unstable input.

Investigation result

The failure is a multi-writer coordination bug made deterministic by a periodic refresh.

Active configuration

The repaired bridge is not active

~/.claude/settings.json currently runs ccstatusline, resolved to /usr/local/bin/ccstatusline. The file ~/.claude/cc-usage-bridge.sh exists, so the app reports “installed,” but Claude Code is not invoking it.

Version split

Two collectors have diverged

/usr/local/bin/ccstatusline is byte-for-byte identical to scripts/ccstatusline: the older, unconditional last-writer version. Commit 273d38a added “keep the higher value” only to the bridge embedded in BridgeInstaller.swift.

Amplifier

Every session writes every 10 seconds

The active setting includes refreshInterval: 10. Official documentation says this reruns the status command on a fixed timer in addition to event-driven updates. Idle sessions therefore keep republishing older snapshots.

Why the values differ

Same account, different snapshot ages

The status-line JSON is session-scoped input. Each open session retains the rate-limit state from its own latest API response. The account window is shared, but sessions do not all receive the newest snapshot simultaneously.

Why the status-bar change triggered it

The change set the configured command back to ccstatusline. That bypassed the app-installed merge bridge while leaving its file behind. The app’s isInstalled check tests file existence only, not whether the active settings point to it.

Live reproduction

A 25-second read-only watch of state.json captured three writers. All rows refer to the same five-hour reset (1783173600), so they are stale views of one account window—not separate limits.

Interpretation

Your earlier 30% → 70% → 55% → 92% sequence is the same signature. By the later sample, the freshest session had advanced to 98–99%; the older sessions continued restoring 55% and 75%.

Known solution

Keep Claude’s official percentage, but stop treating a session callback as exclusive ownership of global state. The most robust design is a tiny per-session event store with aggregation in the app.

1

Write one snapshot per session Recommended

The status command writes atomically to sessions/<session_id>.json. No sessions contend for the same file. The app scans recent snapshots, groups each limit by resets_at, and displays the maximum percentage in the current reset window. Because an account-usage percentage should be monotonic inside one window, the maximum is the least stale observation. When resets_at advances, the app moves to that newer window and permits the percentage to fall.

Claude sessions A, B, C
  ├─ sessions/A.json  55%  reset R
  ├─ sessions/B.json  75%  reset R
  └─ sessions/C.json  99%  reset R
                     ↓ aggregate(reset R) = max
                  menu bar: 99%
2

Locked global merge Smallest patch

Retain one file, but put the complete read–compare–write transaction behind an inter-process lock and use a unique temporary filename. Compare window tuples, not percentages alone: if reset timestamps match, keep the larger percentage; if the incoming reset is newer, replace both fields; if it is older, keep both existing fields. Apply the same rule independently to five-hour and weekly windows.

3

Remove refreshInterval Mitigation only

This stops idle sessions from overwriting the file every ten seconds and is sensible because usage changes only after API activity. It does not eliminate the multi-writer race when several sessions are actively producing responses, so it should accompany—not replace—one of the designs above.

Two necessary cleanup items

Make the app verify that the active statusLine.command actually routes through its collector, and remove the duplicated embedded/standalone implementations (or generate both from one source). Otherwise a future status-line change can silently reintroduce the split.

Acceptance checks for a future fix

No implementation was made in this investigation. These checks define when a later implementation is genuinely done.

  • Feed three session snapshots—30%, 70%, 92%—with one reset timestamp in arbitrary order; the displayed value stays at 92%.
  • Feed a new reset timestamp with 4%; the displayed value drops to 4% and never pairs that value with the previous reset.
  • Run concurrent writers repeatedly; every persisted JSON document remains valid and no newer observation is lost.
  • Leave old sessions idle while one active session advances; periodic rendering cannot lower the menu value.
  • Change the visual status-line renderer; collection remains installed, or the app clearly reports that it has been bypassed.
  • Verify five-hour and seven-day windows independently, including one being absent from a payload.
